The log for V0.7.0.

Target of this version is to extend the calls and configurations for database access. Actually they just execute static configurations in ==>{GithubLinkCall->NONE, EODBR||DbSqlConfig.json}.

2.11.2020

Refactoring the sql configuration part with

  • ==>{ConfigLinkCall->ModelConfig, DbSqlReadCall}. which could be parametrized
  • ==>{ConfigLinkCall->ModelConfig, DbSqlExecuteCall}.

The implementation of database objects with ==>{ConfigLinkCall->ModelConfig, ModelConfigDbObject}. allows to define generic calls with objects.

  • ==>{ConfigLinkCall->ModelConfig, DbModelReadCall}.
  • ==>{ConfigLinkCall->ModelConfig, DbModelWriteCall}.
  • ==>{ConfigLinkCall->ModelConfig, DbModelDeleteCall}.

With this the model configuration could contain permissions or special table names like in ==>{GithubLinkCall->NONE, EOTR||ModelConfig.json}.

The database module should show just the possibilities of the concept: One or several databases could be easily integrated in a generic way. So it's more a "proof of concept" and has no library on mvn central.

What's missing is a object mapping for the request results or a generation of databases. Such features could be integrated with a hibernate module that is in the pipeline.

But productiveness of any database module will need some real projects to fix all potential problems.
