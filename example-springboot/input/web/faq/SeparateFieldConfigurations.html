Why field configurations have separate files?

The fields are independent of the model. In scheme definitions this is also possible.

One can reuse field configuration in model configuration only by its field name. So one field name has exactly one definition. One avoid different field definitions for the same field name in different models.

The EOConfigCache initially load every file named FieldConfig.json from the class path and then resolve the field names in the model configurations by the field configurations.

When the class names within Models.json are resolved, the field entries will be created via reflection using the combination <ClassName> .<FieldName> as naturalId.
