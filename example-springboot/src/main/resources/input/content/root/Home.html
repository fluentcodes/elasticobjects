ElasticObjects is a Framework to build core applications by configurations.

Core of the framework are linked container objects, called Adapter. Each Adapter uses ModelCache objects for object acces, initial loaded from JSON configurations within the classpath.

Several generic actions are implemented using other configuration JSON files in the classpath. The generic actions use also cached configurations, e.g. for database access via hibernate.

It’s written in Java, but the concept is independent of a specific language. An implementation in a non class approach language like JavaScript could save the class generation part and simplify the mapping to the Adapter.

A service example

One of the goals is a service interface as powerful as SOAP without its complexity and verbosity. And the interface should be independent to some webserver like REST.

Compare with a dumb HATEOS-REST solution a client could get all configurations available on the server to create requests and handle responses including typesafety or database constraints.

Request-JSON

The request is in JSON format consists of some data and some actions calling serverside methods reference by the actionKey.

The following request call a read method on a h2 hibernate database with table BasicTest. It filter all entries, where testString has the value “testString3”. The result should be stored in the path “/test”. 

{
  ".actions": [
    {
      "actionKey": "HibernateModelAction.read(h2:mem:BasicTest)",
      "attributes": {
        "where": "testString eq testString3"
      "path": "test"
      }
    }
  ]
}
Response-JSON

The response is filled with result from the database. The info field of the call is filled and furter log informations, when asked for.

State

Actually the service approach is a working proof of concept. It’s comparable small to its power. Even it has almost 1000 tests one can easily create 1000 more due to its range of application.

For the service application field some project experience is needed to make it bullet proofed.

Anyhow the code generation or for data mash things are fields of applications which could save a lot of developing time.

In february I plan to deploy demo application and to push the sources to Gitlab.

Reasons

Code Generation

I’ve started ElasticObjects almost six years ago, when working on a portfolio management project with more than 100 tables and 1500 fields with a relation depth of seven and GBytes of data loaded every nicght from the host. A differential import should be implemented to reduce downtimes.

I’ve created two excelsheet with table data and fields with all the database/hibernate and xml serialization informations including natural ids, since the id is not delivered by the host. Based on these schema informations, we created Java/SQL/Test and documentation via a simple template mechanism.

At the end we reduced the down time for uploading data from 6 hours to half an hour. I was fascinated saving so much time and effort creating Java solutions by a simple extraction of the model structure.

Transportable Service Information

My next project was a Human Resource Management solution. Here a part should refactored with hibernate and JQuery. It had a rather complex access and workflow logic.

During the realization I wished to have the information of the server within the client solution to avoid developing things twice. This was the start of ElasticObjects.
